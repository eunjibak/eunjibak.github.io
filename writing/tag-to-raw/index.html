무엇이 바뀌었나

에이전트를 처음 만들 때는 사용자가 #태그와 감정 점수를 직접 입력하도록 설계했다. 지금은 사용자가 자연어로 아무 말이나 던지면 LLM이 맥락을 읽고 자동 분류한다.

입력 첫 버전:

#에이전트 오늘 기획작업을 마쳤다 🐱🐱🐱🐱

현재 버전:

오늘 피그마 컴포넌트 정리 끝!

→ 시스템이 알아서: action=실행, emotion=기쁨, north_star=agentcat 으로 분류.

왜 이게 어려웠나: 디자이너에게 특히

개발 경험 없이 LLM의 도움만으로 첫 버전을 만들었다. 기존의 워크플로우대로 기획서를 적고, 기존의 회고 입력을 기반으로 구현에 성공했지만 이상하게 불편했다. 한 문장으로 말하면 알아서 분류해주면 될 일을, 왜 내가 태그를 고르고 이모지 개수를 세고 있지? 그 지점에서 처음으로 내가 짜놓은 구조가 잘못됐다는 걸 의심하게 됐다.

왜 이 사고 전환이 특히 디자이너에게 어려운지, 만들고 나서야 보였다.

기존 UX 사고에이전트 UX 사고
사용자 → 인터페이스 → 결과사용자 → 상태 → 판단 → 개입 → 패턴 변화

어디서 이탈하는가?사용자의 패턴은 무엇인가?
정해진 필드에 사용자가 맞춰 입력사용자가 던진 말을 시스템이 해석

디자이너는 늘 완성된 화면에서 역으로 기초 구조를 짜는 습관이 있다. 사용자 눈에 마지막으로 닿는 장면을 먼저 그리고, 거기서 역으로 플로우와 데이터 구조를 설계한다. 이 관성이 에이전트를 만들 때도 그대로 작동했다.

텔레그램에서 태그 선택, 이모지로 감정 점수, 스프레드시트 특정 열에 값이 들어가는 흐름. 당시엔 이걸 사용자 입장에서 잘 설계된 흐름이라고 믿었다. 확정적인 UI 흐름에 사고가 갇혀 있었기 때문이다.

어떻게 전환했나

  • 입력단: #태그와 이모지 수치에서 자연어 한 문장으로
  • 해석단: 사용자가 기입한 드롭다운 값에서 LLM이 문맥에서 추출한 태그로 (action / direction / emotion / north_star)
  • 출력단: 고정 컬럼 조회에서 필요할 때 필요한 데이터만 조합해서 답변으로

중요한 변화는 단순한 입력 간소화가 아니다. 인간이 분류해서 시스템에 넣어주는 것에서 인간이 raw 상태로 던지고 시스템이 분류하는 것으로, 책임의 주체가 바뀐 것이다.

왜 이게 디자이너에게 중요한 인사이트인가

이 전환을 받아들이면 디자이너의 일이 근본적으로 달라진다.

에러 메시지를 쓰는 일시스템이 정중하게 되묻는 정책을 설계하는 일
정해진 형식대로 입력하세요무엇이든 던지세요, 제가 해석하겠습니다

기존 UX가 Input Validation(입력 제한) 중심이었다면, 에이전트 UX는 Intent Interpretation(의도 해석) 중심이다. 입력을 제한해서 에러를 줄이는 대신, 의도를 해석해서 맥락을 맞춘다.

예시: 인식의 확장

이 사고 전환은 에이전트에만 적용되지 않는다. 모든 필드를 필수로 채워야 답을 얻고, 1 선택, 2 선택, 다음, 확인으로 진행되는 기존 서비스의 워크플로우 전체가 사용자 관점에서 이미 무거워지고 있다.

사용자는 이미 맥락이 빠진 자연어로 말해도 숨겨진 의도까지 추론해서 제안받는 경험을 하고 있다. 확정적인 화면과는 다른 흐름이 이미 도래한 것이다.

단순 개인화를 넘어, 원석과 재료로 현재의 나에게 맞게 늘 가공하는 것. 그게 raw 기반 UX의 본질이다.
