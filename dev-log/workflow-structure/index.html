이 워크플로우의
구조

이 프로젝트에서 만든 건 회고 에이전트 하나지만, 만들고 보니 그 안에 회고에만 한정되지 않는 골격이 있더라구요. 입력에서 학습까지 다섯 단계로 정리되는 구조인데, 도메인을 바꿔서도 옮겨갈 수 있을 것 같았어요.

5단계 틀로 정리되는 구조

이 프로젝트에서 만든 건 개인 회고 에이전트지만, 만들고 보니 그 안에 회고에만 한정되지 않는 골격이 있다는 걸 알게 됐어요. 5단계로 정리하면 이렇게 돼요.

1. 입력 (Input)     : 사용자가 자연어로 던진 raw 데이터
2. 상태 (State)     : 최근 N일 데이터로 현재 상황 계산
3. 정책 (Policy)    : 언제 말하고 언제 침묵할지 룰
4. 실행 (Execution) : 상태 + 정책 조합으로 개입 수행
5. 학습 (Learning)  : 주기적 패턴 분석 후 정책 조정

이 다섯 단계가 한 사이클을 이루면서 매일 반복돼요. 사용자의 행동 데이터가 쌓이고, 그 데이터를 기반으로 판단이 필요한 영역이라면 회고가 아닌 다른 도메인에서도 같은 골격이 작동할 수 있다고 느꼈습니다.

다음 도메인의 워크플로우 실험

회고 다음에는 같은 5단계 틀을 다른 개인 영역에 적용해보려고 해요. 도메인만 갈아 끼우면 다른 데이터·다른 정책이 들어오고, 그래도 입력 → 상태 → 정책 → 실행 → 학습이라는 사이클은 그대로 유지될 거라고 보고 있어요.

지금 머릿속에서 굴리고 있는 후보는 두 가지예요. 하나는 소비 데이터를 감정과 연결해서 읽어주는 가계부 에이전트, 다른 하나는 학습과 시도 사이의 간극을 추적하는 성장 기록 에이전트. 둘 다 매일 상태가 바뀌고, 그 상태를 스스로 감지하기 어려운 영역이라는 공통점이 있어요. 숫자로만 보면 안 보이고, 직관만으로는 놓치는 영역이라는 점에서 회고와 닮았습니다.

앞으로의 계획은 이 5단계 틀을 뼈대 삼아 다른 도메인의 워크플로우 실험을 하나씩 이어가는 거예요. 회고 에이전트에서 검증한 구조가 다른 영역에서도 작동하는지를 직접 만들어보면서 확인해보려고 합니다.
